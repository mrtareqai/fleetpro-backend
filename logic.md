# منطق وعمليات نظام FleetPro (Business Logic & Workflow)

يوضح هذا الملف المنطق البرمجي والعمليات التشغيلية (Business Logic) لنظام **FleetPro**، بدءاً من دور المالك الرئيسي (TopOwner) وحتى عملية حجز التذاكر وإدارة الرحلات والسيارات.

---

## 1. المعمارية الموحدة (Multi-Tenancy)
النظام مبني كـ **SaaS (Software as a Service)**، حيث يمكن لعدة شركات نقل (Tenants) استخدام نفس التطبيق ونفس قاعدة البيانات، ولكن مع **عزل تام للبيانات**.
- **المعرف الأساسي `tenantId`**: كل شركة لها `tenantId` فريد، وأي عملية في النظام (إضافة سيارة، رحلة، مستخدم) ترتبط بهذا الـ `tenantId`.
- **العزل (Isolation)**: أي مستخدم عادي لا يمكنه رؤية أو تعديل سوى البيانات الخاصة بـ `tenantId` شركته.

---

## 2. دور المالك الرئيسي (TopOwner)
- **الصلاحيات القصوى**: هو المستخدم السوبر (مثل مدير المنصة). 
- الـ `tenantId` الخاص به يكون `null` أو مخصص ليمنحه صلاحية رؤية كل شيء.
- **إدارة الشركات**: 
  - يمكنه إنشاء شركات جديدة (Tenants).
  - يمكنه إيقاف/تفعيل اشتراك الشركات (`isActive` toggle).
- **الإحصائيات الشاملة**: لديه لوحة تحكم (Dashboard) تعرض إحصائيات على مستوى النظام بالكامل (إجمالي التذاكر، الدخل، الخ).
- **الجدار الناري وسجل النظام (Firewall & Audit Logs)**: يراقب الـ IPs المحظورة والنشاطات الأمنية على مستوى الخوادم.

---

## 3. دورة حياة الرحلات والتذاكر (Workflow)
العملية التشغيلية داخل كل شركة تسير بشكل متسلسل ومترابط:

### أ. الإعداد المسبق (Setup)
قبل إطلاق أي رحلة، يجب على الإدارة تجهيز التالي:
1. **المركبات (Vehicles)**: إضافة سيارات الشركة (الموديل، رقم اللوحة، تاريخ الفحص).
2. **السائقين (Drivers)**: إضافة بيانات السائق وتاريخ انتهاء الرخص.
3. **المسارات (Routes)**: تعريف خطوط السير الثابتة (مثل: الرياض -> جدة)، تحديد السعر الأساسي (`basePrice`)، ومدة الرحلة التقريبية.
4. **الفروع والوكلاء (Branches & Agents)**: (اختياري) تعيين فروع المحطة أو وكلاء الحجز، وتحديد العمولات الخاصة بهم.

### ب. إنشاء الرحلة (Trips Creation)
- يقوم مشرف العمليات بجدولة **رحلة (Trip)** لتاريخ ووقت محددين.
- **ربط البيانات**:
  - يختار **المسار (Route)**: ليعرف وجهة الرحلة.
  - يختار **المركبة (Vehicle)**: لتحديد عدد المقاعد الكلية (`totalSeats`).
  - يختار **السائق (Driver)**: لتكليفه بالرحلة.
- **المقاعد المتاحة (`availableSeats`)**: تكون في البداية مساوية لعدد المقاعد الكلية في السيارة المختارة.
- **الحالة (Status)**: تكون الرحلة في البداية "مجدولة" (Scheduled).

### ج. حجز التذاكر (Bookings / Tickets)
عندما يأتي العميل لحجز مقعد:
1. يختار الوكيل أو الموظف **الرحلة المجدولة**.
2. يتحقق النظام من **المقاعد المتاحة (`availableSeats > 0`)**.
3. **يتم إصدار التذكرة**: وتسجيل (رقم التذكرة، اسم الراكب، رقم المقعد، السعر).
4. **تحديث الرحلة**: بمجرد الدفع وتأكيد الحجز، ينقص عدد المقاعد المتاحة في الرحلة بمقدار 1 (`availableSeats - 1`).
5. **المعاملات المالية (Transactions)**: يُسجل دخل مالي (Income) في النظام لصالح الـ `tenantId`.

### د. تسيير الرحلة (Trip Execution & Movements)
1. في وقت الانطلاق، تتغير حالة الرحلة إلى **"جارية" (In Progress)**.
2. يتم تتبع **حركات الرحلة (Movements)**: (انطلاق، توقفات، وصول).
3. يمكن ربط **مصروفات (Expenses)** بالرحلة (مثل: وقود، صيانة طارئة) وتُخصم من إيرادات الرحلة لتحديد الأرباح.
4. عند الوصول للوجهة النهائية، تتغير حالة الرحلة إلى **"مكتملة" (Completed)**.

---

## 4. ملخص العلاقات في قاعدة البيانات (Entity Relationships)
لتحقيق المنطق السابق، ترتبط جداول قواعد البيانات كالتالي:

- `User` ⬅️ ينتمي لـ `Tenant` (إلا الـ TopOwner).
- `Route` ⬅️ ينتمي لـ `Tenant` + يمكن أن يرتبط بـ `Agent` و `Branch`.
- `Vehicle` ⬅️ تنتمي لـ `Tenant`.
- `Driver` ⬅️ ينتمي لـ `Tenant`.
- `Trip` ⬅️ ترتبط بـ (`Tenant`, `Route`, `Vehicle`, `Driver`).
- `Ticket` ⬅️ ترتبط بـ (`Tenant`, `Trip`).
- `Expense` / `Transaction` ⬅️ ترتبط بـ (`Tenant`, `Trip` أو `Vehicle` أحياناً).

---

## 5. إدارة الحالات (State Management) والـ API
- تم استخدام حزمة **Riverpod (StateNotifierProvider)** داخل تطبيق الـ Flutter.
- يتم إرسال طلبات الـ API عبر **Dio** (من خلال `ApiService`).
- في كل طلب يتم تلقائياً إرسال الـ Token (مستقبلاً) والـ `tenantId` للـ Backend لضمان جلب البيانات المخصصة للشركة التي ينتمي لها الموظف فقط.
- **Error Handling**: إذا حصل خطأ في الاتصال، يتم تحويل حالة التطبيق لـ Error State و تظهر رسالة خطأ مع خيار "إعادة المحاولة"، مما يحافظ على استقرار النظام (UX).
